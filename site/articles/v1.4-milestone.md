# v1.4 Milestone — From Observation to Memory and Trustworthy Delivery

> v1.0 narrowed the surface. v1.2 enriched each response with time and causality. The v1.3 and v1.4 series go two further steps: they let the agent re-query its past calls, and they close the gaps where a call could appear to succeed without actually doing anything.

## Where v1.2 left things

v1.2 made each response carry its own sense of time, confidence, and causal context — opt-in via `include`. An agent could ask "what just changed because of my click?" and read the answer in a single call instead of inferring it from screenshot diffs.

That solved a single-call problem: knowing what is true *now*, in reference to the most recent action.

It still left two questions on the table:

- **What about the call before that?** An agent working through a multi-step recovery had no way to look at its own past episodes — the responses for those calls had already been consumed, and there was no projection of "what I have just done" available on the next call.
- **Did my call actually do anything?** Some delivery channels — `WM_CHAR` to a foreign window, a CDP click against a button with no listener attached, a fill against a controlled input that silently transformed the value — could return `ok:true` while having no observable effect. The agent had to invent its own verification each time.

The v1.3 and v1.4 releases address those two questions directly.

## What v1.3 adds — a memory the agent can re-query

v1.3 turns the implicit "the agent has just made a tool call" into four explicit, queryable layers, all opt-in via `include`.

### Working memory — what I just did

```
include: ["working:5"]
```

Returns a compact summary of the most recent five tool calls (default 10, max 50), including in-flight calls. Each entry is a small fixed shape: tool call id, tool name, truncated args, ok flag, compound-boundary flag.

Useful for "remind me what I just did before I run this macro" — the kind of question an agent currently answers by re-reading a transcript it may not have any more.

### Episodic memory — the rich detail of past calls

```
include: ["episodic:3"]
```

Returns the most recent three completed tool calls in rich form: lease-token summary, started/completed event ids, elapsed time, longer-truncated args. Default 5, max 100.

Different from working memory in two ways: it is rich-form rather than compact, and it skips in-flight calls. The use case is "show me how a similar successful call was timed before I fix this failure" — a step that previously required re-running the call to gather telemetry.

### Semantic memory — patterns this window already accepts

```
include: ["semantic:3"]
```

Returns rule-based extractions of "in this window, this sequence of three or more tool calls all succeeded". Default 3, max 10. Patterns are stored in an in-memory LRU and can optionally be persisted to disk between sessions.

This layer is intentionally small and conservative: only fully-successful sequences become patterns, and only on the same window title. The point is to give the agent a starting hint for repeated workflows without claiming to be a planner.

### Procedural memory — workflows worth re-running

```
include: ["procedural:3"]
```

Returns aggregated `run_macro` outcomes that have succeeded at least three times, never failed, and contain no destructive tools.

The "no destructive tools" filter is deliberate. Macros that include `mouse_click`, `keyboard`, `terminal`, `clipboard`, `browser_*`, `workspace_launch`, or similar are excluded from suggestions even if they have run successfully many times. Auto-suggesting destructive workflows is out of scope at this layer; an explicit-consent flow can be added later if needed.

### Per-call security tier

Memory storage is gated by two axes: an operator-controlled environment ceiling (whether persistence and redaction are on), and a per-call request via `include: ["memory_strict" | "memory_balanced" | "memory_open"]`.

The rule is one-directional: an LLM can ask for *more* security than the operator default (`memory_strict` forces redaction on and procedural suggestions off, regardless of env), but it cannot ask for *less*. If the operator set persistence off, no per-call value can turn it back on.

### A worked example

```
desktop_state({ include: ["envelope", "causal", "working:5"] })
```

This single call now returns: the current desktop state, when it was observed, how confident the engine is, what the agent's last action did, *and* a compact summary of the last five things the agent has done. Five questions, one round-trip.

## What v1.4 adds — closing the silent-failure gaps

v1.4 is a series of patches that tighten the "did my call actually do anything?" half of the loop. Three themes run through it.

### Typed error codes for refusal paths

Auto-guard refusals (ambiguous target, target not found, blocked by modal, browser not ready, identity changed, and so on) used to surface as a generic `ToolError`. Callers had to parse the message string to know what to do next. v1.4.0 introduces `code: "AutoGuardBlocked"` with a status enum and matching recovery hints, so agents can branch on the typed code instead.

Spawn-time refusals from `workspace_launch` and `browser_open` now surface as `code: "SpawnFailed"` with a small set of typed recovery hints (full path, permission, elevation, built-in command, Windows policy). The `ENOENT` / `EACCES` / `EPERM` distinction the OS reports is preserved in the suggest text.

### Partial-failure reporting

`run_macro` with `stop_on_error: false` used to require the caller to JSON-parse each step's nested response to find failures — an awkward inversion of the macro abstraction. v1.4.0 adds a top-level `warnings[]` array that lists any step whose nested tool returned `ok:false`, alongside the step number, the tool name, and the typed code if available. Backwards-compatible: the field is omitted when no step failed.

A sibling fix: `stop_on_error: true` now actually stops on a nested `ok:false`. Previously a step whose handler returned an `ok:false` envelope still counted as `ok:true` at the macro level (the handler did not throw), so the macro continued running and could corrupt downstream state.

### Delivery verification

This is the largest of the three themes.

- **`browser_click`** installs a `MutationObserver` on the page before the click and inspects DOM mutations, URL changes, and `document.activeElement` shifts over a 500 ms window. The response carries `hints.verifyDelivery.status = "delivered"` if any signal is observed and `"unverifiable"` (with a typed reason) otherwise. SPA buttons with no listener attached used to silently absorb the click; they now report it.
- **`browser_fill`** reads the input's `value` back after dispatch and fails with `BrowserFillNotDelivered` if the result does not match the request. Controlled inputs that transform the value (numeric filters, max-length, format masks) are distinguished from outright rejection so the caller can treat the actual value as authoritative when appropriate.
- **`terminal({action:'send', method:'background'})`** auto-detects hidden-input prompts (such as `password:`, `passphrase:`, `sudo`, `Read-Host -AsSecureString`) and reports `unverifiable` instead of falsely reporting `BackgroundInputNotDelivered`. The keystrokes were always reaching the prompt; the read-back loop just could not see the echo.
- **`keyboard({action:'type', method:'background'})`** to Windows 11 New Notepad now reports `delivered` instead of `unverifiable`. Notepad's editor element exposes `ValuePattern` but not `TextPattern`; the verification path now falls back to `ValuePattern` correctly.

### A new opt-in channel for Windows Terminal

`method: 'background'` had been refused on Windows Terminal since v1.3.2 because the WinUI/XAML pipeline silently drops `WM_CHAR`. v1.4.2 introduces an opt-in alternative: `method: 'foreground_flash'` briefly steals foreground, pastes the text via the clipboard, and restores the previous foreground in roughly 50–200 ms. The clipboard is saved and restored across the operation, with explicit hints about the typing-leak risk during the flash window.

The default `method: 'background'` continues to refuse Windows Terminal targets — the silent-fail guarantee from v1.3.2 is unchanged. `foreground_flash` is purely additive.

### Patch series, not a single release

The v1.4 work landed across v1.4.0, v1.4.1, and v1.4.2. v1.3.x also had two patches of its own (v1.3.1 fixed a regression where six tools rejected all input on v1.3.0; v1.3.2 fixed the Windows Terminal silent-fail). The full sequence is in the changelog.

## Why these two together

v1.3 gives the agent a memory of its own past calls. v1.4 gives the agent confidence that those past calls actually did what they reported.

Each is independently useful. Together they close the same loop from two ends: the agent can both look back at what it did *and* trust the verdict on whether it worked.

## Conclusion

v1.0 narrowed the surface so each call meant more. v1.2 enriched each response so a single call could answer "is this current?" and "did this happen because of me?". v1.3 made the agent's own past calls a queryable resource. v1.4 closed the gaps where a call could appear to succeed while doing nothing.

The direction is consistent: every release pushes a little further from "infer, hope, retry" toward "observe, verify, decide".
