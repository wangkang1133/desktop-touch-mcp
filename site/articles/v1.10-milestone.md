# v1.10 Milestone — Targeting by Meaning, Confirming in One Call

> v1.0 narrowed the surface. v1.2 deepened each response. v1.4 made delivery trustworthy, and v1.5–v1.8 carried that through to completion. The v1.9–v1.10 pair turns to *targeting* and *confirmation*: it lets you name a target by what it means on the two surfaces that fought hardest against that idea — the browser DOM, and the apps that expose no accessible structure at all — and it folds "confirm the result, find the next target" back into the act itself.

## Where v1.8 left things

By v1.8 an agent could trust the act-and-observe loop: a click reached a live target, a command's completion and exit code were known, and visual verification no longer raced. Two friction points remained, at opposite ends of the spectrum.

- **On the browser, targeting was still positional.** `browser_click` and `browser_fill` took a CSS selector — a brittle, structural path that breaks the moment the markup shifts. The DOM is the richest, most semantic surface there is, yet the agent addressed it by location, not by meaning.
- **On a "visual-only" target, confirming an act meant leaving the act.** When you act on something the accessibility tree cannot describe — an Electron or PWA app, a game, a custom-drawn canvas, a Remote Desktop window — there are no entities to diff. To learn what the click did, the agent had to follow every `desktop_act` with a separate `desktop_state` and a `screenshot`, then re-`desktop_discover` to find the next target. Three round-trips to confirm one click.

v1.9 and v1.10 close both — and they close them with the *same* idea the project has pushed since v1.0: address the interface by meaning, and let each step carry the evidence the next one needs.

## What v1.9 adds — semantic targeting reaches the browser

v1.9 lets the browser tools name an element by what it *means* instead of where it sits in the markup:

```
browser_click({ by: "text", pattern: "Sign in" })
browser_fill({ by: "ariaLabel", pattern: "Search", value: "mcp" })
```

`by` accepts `text`, `role`, or `ariaLabel`; `pattern` is what to match. The resolver gathers every candidate, and if exactly one matches it acts — if several match, it **stops and reports the ambiguity** rather than clicking a guess. This is the browser equivalent of a lease: commit to a single, identified target or decline.

The same release teaches the browser to tell a real modal dialog from an ordinary navigation drawer. `browser_overview` reports a machine-readable `modal` state, and `browser_click` refuses to click *through* a blocking dialog onto its backdrop — the web version of the modal-blocking guard the native path already had.

It is "see entities, not coordinates," finally applied to the surface that always had the entities — the DOM — but had only ever been addressed by selector.

## What v1.10 adds — a visual-only act that confirms itself

v1.10 turns to the other extreme: the targets with *no* structure to address. When you act on a visual-only target, a successful `desktop_act` can now bundle a **`roiCapture`** straight into its response:

```
desktop_act({ lease, action: "click", returnCapture: "on-change" })
// → { ok: true, diff: [...],
//     roiCapture: { somImage: "<PNG of the region that changed>",
//                   entities: [ /* lease-less preview of what's now there */ ],
//                   roi, source } }
```

- **`somImage`** is a base64 PNG cropped to *just the region that changed* — not the whole window — so you can see the result of your action.
- **`entities`** is a lightweight, read-only preview of the labels and controls now visible in that region — the next targets, surfaced without a separate discover.
- **`roi` / `source`** say where the crop came from.

That folds the old three round-trips — act, then `desktop_state`, then `screenshot` — into **one call**. The agent confirms what its click did and sees the next target in the same response it already had to read.

A `returnCapture` option controls it: `"on-change"` (the default for visual-only targets) attaches the capture only when the screen actually changed; `"always"` attaches it on any successful act; `"never"` suppresses it. The preview `entities` carry no lease — they are a preview, not a handle — so you re-run `desktop_discover` to act on one, exactly as before.

Two properties keep it honest. It is **scoped to visual-only targets**: on a structured target (a browser tab over the DevTools Protocol, an accessibility-rich native window) no `roiCapture` is attached, because there `desktop_state` and `desktop_discover` are cheaper and exact — spending pixels there would be a regression, not a feature. And the act's **semantic `diff` stays correct** even though the changed region is re-read visually: a stable on-screen label is never reported as having vanished, because the diff is judged against what was already discovered rather than re-read from a tight crop.

## Why these together

v1.9 and v1.10 are the same move aimed at the two surfaces that resisted it longest:

- **Target by meaning, everywhere** — a browser element by its text/role/label, a native entity by its lease. Both commit to one identified target or decline; neither guesses.
- **Carry the evidence forward** — a blocking-dialog state the browser can act on, and a changed-region crop plus next-target preview the visual-only act returns inline. The step that acts hands the next step what it needs.
- **Spend perception only where it pays** — the heavy visual capture appears only on the targets that have no cheaper truth, and only when something actually changed.

The direction is unchanged from every release before: a little further from "infer, hope, retry" toward "observe, verify, decide" — now reaching the browser DOM on one end and the structureless canvas on the other.

## Conclusion

v1.0 narrowed the surface so each call meant more. v1.2 deepened the response, v1.3 gave it memory, v1.4 made delivery trustworthy, and v1.5–v1.8 carried that through to completion. v1.9–v1.10 extend the founding idea — *see entities, not coordinates* — to the two surfaces that had always been addressed by location: the browser, now targeted by meaning, and the visual-only canvas, where a single act now confirms its own result and points at what to do next.

> These tools are experimental and Windows-first; the World-Graph surface (`desktop_discover` / `desktop_act`) and the browser semantic targeting described here are evolving release to release.
