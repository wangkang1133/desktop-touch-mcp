## 1. Goal

Windows 11 24H2+ の **新 Windows ML API** (`Microsoft.Windows.AI.MachineLearning.ExecutionProviderCatalog`) を Rust から呼び、ハードウェア互換な全 EP (DirectML / AMD Ryzen AI XDNA NPU / Intel OpenVINO / Qualcomm QNN 等) を ORT に dynamic 登録できる状態にする。`ep_select` cascade の **Layer 1 (最優先)** として統合し、WinML が成功すれば以降の DirectML / ROCm / CUDA attempt を skip する。

単一目標:

> `windows-app` crate 経由で `ExecutionProviderCatalog::GetDefault()` → `FindAllProviders()` → 各 `ExecutionProvider::EnsureReadyAsync()` + `register_execution_provider_library(name, library_path)` を実行し、その後 `ort::Session` を作成して「WinML(<space-separated EP names>)」ラベルで cascade に選ばれる。

このバッチで **NPU 推論の道が開く** (AMD Ryzen AI / Intel NPU / Qualcomm Hexagon 等、すべて ExecutionProviderCatalog 経由で自動登録)。これは ADR-005 D7' vendor portability の本丸。

## 3. API design

### 3.1 `src/vision_backend/winml.rs` (feature = "vision-gpu-winml" 時のみ compile)

```rust
//! WinML integration — new Windows ML API (Microsoft.Windows.AI.MachineLearning).
//!
//! Wraps `ExecutionProviderCatalog` via the `windows-app` crate (WinRT bindings
//! generated by windows-bindgen from Microsoft.WindowsAppSDK.ML.winmd). The
//! `register_all_eps_to_ort` helper runs the full "enumerate → ensure ready →
//! register library" flow once per session init.
//!
//! Only compiled with `--features vision-gpu-winml` on Windows 11 24H2+
//! (build 26100) systems. On older Windows the feature flag is OFF by default
//! and `winml_attempt` returns the stub Err (see `winml_fallback.rs`).

use crate::vision_backend::error::VisionBackendError;

/// Enumerate compatible EPs via ExecutionProviderCatalog and register each
/// with the ORT environment. Returns the list of registered EP names
/// (e.g. ["DirectML", "XDNA", "OpenVINO"]).
///
/// Errors:
/// - ExecutionProviderCatalog::GetDefault failure (WinAppSDK not bootstrapped)
/// - FindAllProviders returns empty (no EPs available on this hardware)
/// - EnsureReadyAsync fails (download blocked, network off, license decline)
/// - register_execution_provider_library fails (library path invalid)
///
/// The function is synchronous from the caller's perspective — internally
/// awaits `IAsyncOperation<bool>` via `.get()` which blocks the worker thread.
/// Safe because `init_session_blocking` already runs on a libuv worker
/// (`VisionInitSessionTask::compute`), not the V8 main thread.
pub fn register_all_eps_to_ort() -> Result<Vec<String>, VisionBackendError> {
    use windows_app::Microsoft::Windows::AI::MachineLearning::ExecutionProviderCatalog;

    let catalog = ExecutionProviderCatalog::GetDefault()
        .map_err(|e| VisionBackendError::Other(format!("GetDefault failed: {e}")))?;

    let providers = catalog.FindAllProviders()
        .map_err(|e| VisionBackendError::Other(format!("FindAllProviders failed: {e}")))?;

    let mut registered: Vec<String> = Vec::new();

    for provider in providers.into_iter() {
        let name_hs = provider.Name()
            .map_err(|e| VisionBackendError::Other(format!("provider.Name: {e}")))?;
        let name: String = name_hs.to_string_lossy();

        // EnsureReadyAsync — downloads the EP package if missing. IAsyncOperation<bool>.
        // .get() blocks until the operation completes, returns bool.
        let ready_op = provider.EnsureReadyAsync()
            .map_err(|e| VisionBackendError::Other(format!("EnsureReadyAsync start: {e}")))?;
        let ready: bool = ready_op.get()
            .map_err(|e| VisionBackendError::Other(format!("EnsureReadyAsync wait: {e}")))?;
        if !ready {
            // Provider refused to prepare (license etc.) — skip rather than bail out.
            continue;
        }

        let lib_hs = provider.LibraryPath()
            .map_err(|e| VisionBackendError::Other(format!("provider.LibraryPath: {e}")))?;
        let library_path: String = lib_hs.to_string_lossy();

        register_ort_library(&name, &library_path)?;
        registered.push(name);
    }

    if registered.is_empty() {
        return Err(VisionBackendError::Other(
            "WinML catalog returned no usable EPs on this machine".into(),
        ));
    }
    Ok(registered)
}

/// Call `ort::Environment::register_execution_provider_library(name, library_path)`
/// if the ort crate exposes it, otherwise fall back to the raw OrtApi.
///
/// ort 2.0.0-rc.12 may or may not expose this in its safe Rust surface — Sonnet
/// checks at implementation time and picks the best available path:
///
/// Path A (preferred, if available): `ort::env()?.register_execution_provider_library(...)`
/// Path B (fallback): Call `OrtApi::RegisterExecutionProviderLibrary` via
///   `ort::api()` and the raw C struct pointer.
///
/// This function centralises the choice so callers only see a single API.
fn register_ort_library(name: &str, library_path: &str) -> Result<(), VisionBackendError> {
```

### 3.3 `src/vision_backend/ep_select.rs` の `winml_attempt` 置換

```rust
fn winml_attempt() -> EpAttempt {
    EpAttempt {
        kind: SelectedEp::WinML { registered_eps: Vec::new() },
        apply: std::sync::Arc::new(|builder| {
            // Register all available EPs via WinML catalog BEFORE building the session.
            // The registered_eps list is captured as a side effect into the EpAttempt
            // kind via the shared state pattern below (SelectedEp mutation is not
            // allowed post-construction — see §3.4 for the workaround).
            #[cfg(feature = "vision-gpu-winml")]
            {
                use crate::vision_backend::winml;
                let eps = winml::register_all_eps_to_ort()
                    .map_err(|e| ort::Error::new(format!("WinML register_all: {e}")))?;
                // Stash registered EPs via thread_local so VisionSession::create can
                // read them back for the SelectedEp::WinML { registered_eps } label.
                WINML_LAST_REGISTERED.with(|cell| *cell.borrow_mut() = eps);
                // Return builder unchanged — session uses the now-registered EPs
                // automatically when `commit_from_file` runs.
                Ok(builder)
            }
            #[cfg(not(feature = "vision-gpu-winml"))]
            {
                Err(ort::Error::new("WinML EP disabled at build time (vision-gpu-winml off)"))
            }
        }),
    }
}

#[cfg(feature = "vision-gpu-winml")]
thread_local! {
    static WINML_LAST_REGISTERED: std::cell::RefCell<Vec<String>> =
        std::cell::RefCell::new(Vec::new());
}
```

## 4. EP / モデル統合の具体仕様

### 4.1 EP cascade 順序 (更新後)

| profile | features | 試行順序 |
|---|---|---|
| Win11 24H2 + AMD RDNA4 | default + `vision-gpu-winml` | **WinML (auto-registers DirectML + XDNA)** → DirectML(0) → CPU |
| Win11 24H2 + Intel + NPU | default + `vision-gpu-winml` | **WinML (auto-registers DirectML + OpenVINO NPU)** → DirectML(0) → CPU |
| Win11 23H2 以下 | default + `vision-gpu-winml` | WinML attempt fails → DirectML(0) → CPU |
| default build (no winml feature) | default | (Phase 4b-1 と同じ) DirectML(0) → CPU |

### 4.2 WinML が成功した後の Session 作成

`register_all_eps_to_ort` で ORT 環境に EP library を登録した後、通常の `Session::builder().commit_from_file(model_path)` を呼ぶと ORT が自動で最適 EP を選択する。

- 我々の「1 EP ずつ試行」cascade 思想とは少し違うが、WinML の利点は「複数 EP を自動 discovery + register」なのでこのまま活かす
- Session 作成後、どの EP が実際に使われたかは ORT の session info から取得可能 (Sonnet 実装時に `session.get_providers()` 等で確認し、log に出力)

### 4.3 WinAppSDK Bootstrap

`Microsoft.Windows.AI.MachineLearning` は WinAppSDK の framework-dependent deployment。実行時に Bootstrap API で activate する必要あり:

- `windows-app` crate の `DynamicDependency::Bootstrap::initialize()` を起動時 1 回呼ぶ
- 呼ばないと `ExecutionProviderCatalog::GetDefault()` が HRESULT error で失敗
- 場所: `init_session_blocking` の dylib init の直後が自然

```rust
pub fn init_session_blocking(init: NativeSessionInit) -> NativeSessionResult {
    if let Err(e) = dylib::ensure_ort_initialized() { return err_result(e); }

    #[cfg(feature = "vision-gpu-winml")]
    if init.profile.winml {
        if let Err(e) = winml::ensure_winappsdk_bootstrapped() {
            // Not fatal — log and continue, cascade will try non-WinML EPs.
            eprintln!("[vision-gpu] WinAppSDK bootstrap failed: {e}");
        }
    }

    // ... 既存の session::VisionSession::create 呼び出し ...
}
```

### 4.4 windows-app crate に WinML API が未露出の場合の対処

最新 `windows-app` (0.4.x 以上) でも WinML API が含まれていない可能性あり。その場合の対処:

**Path C (自前 binding 生成)**:
- `build.rs` で `windows-bindgen` を実行し、`Microsoft.WindowsAppSDK.ML.winmd` から Rust binding を生成
- winmd は NuGet package `Microsoft.WindowsAppSDK.ML` に同梱
- `scripts/fetch-winml-winmd.mjs` を新規作成し、NuGet からダウンロード + extract

Path C を選ばざるを得ない場合、Sonnet は handbook §5 stop condition (設計書にない判断が必要) を発動 → Opus 委譲。Opus が本設計書を revise する。

### 4.5 panic isolation (L5)

- `register_all_eps_to_ort` は `Result` を返す、panic しない
- `ExecutionProvider::EnsureReadyAsync().get()` が失敗しても該当 provider を skip するだけ (全体はそのまま続行)
- WinAppSDK Bootstrap が失敗しても WinML attempt を skip、cascade の次に進む

## 7. Known traps

### Phase 4a / 4b-1 で観測した罠 (再発させない)

1. windows crate 0.62 API mismatch — WinML API は windows-app (別 crate) 経由なので直接は影響なし、ただし WinRT の HSTRING / IInspectable 扱いは共通、`to_string_lossy()` の正しい使い方注意
2. ort prebuilt 不在 (msys2/gnu) — 既に load-dynamic 採用、影響なし
3. Cargo.toml の features 漏れ — `vision-gpu-winml = ["vision-gpu", "dep:windows-app"]` で `dep:` prefix 必須

### 4b-2 で予想される罠

| 罠 | 対策 |
|---|---|
| **windows-app crate に WinML API が未露出** | Path C (windows-bindgen で自前生成) を Sonnet が判定。OPERATOR: 未露出なら設計書 §4.4 に従い Opus 委譲 |
| WinAppSDK Bootstrap 未呼び出しで `GetDefault()` が HRESULT 0x80040154 (Class not registered) | `ensure_winappsdk_bootstrapped` を init_session_blocking の開始時に呼ぶ |
| `IAsyncOperation<bool>::get()` が UI thread 前提で blocking 禁止 (MTA vs STA) | libuv worker は STA ではない、`.get()` 安全。ただし windows-app の bootstrap で `CoInitialize` の apartment 指定が必要な場合あり |
| `ort::Environment::register_execution_provider_library` が ort 2.0.0-rc.12 に無い | Path B (raw OrtApi 経由) — `ort::api().RegisterExecutionProviderLibrary` 呼び出し。Sonnet が動作確認して Path A/B 選択 |
| EP library_path が UNC path で file: URI でない | `&str` で渡して PathBuf::from() で正規化 |
| EP download が数 GB サイズで `EnsureReadyAsync` が数十分 | 本 batch では timeout 設定なし、長時間待ち許容。将来 4b-7 で progress callback 検討 |
| `EnsureAndRegisterCertifiedAsync` (簡易版) で済ませる選択肢もある | 設計書では `FindAllProviders` + 個別 register の方を採用 (registered EP 名を個別に取得するため) |
| `windows-app` の Bootstrap が process 単位で 1 回のみ | `OnceLock` で gate (dylib.rs と同パターン) |
| XDNA NPU 対応で Ryzen AI Software (Vitis) 追加依存? | `EnsureReadyAsync` で自動 download される想定。ユーザー環境で Ryzen AI 拡張パッケージ install 済か手動確認必要 |
| WinML が DirectML を register した後、cascade の DirectML attempt が重複する | WinML 成功時は cascade 打ち切り (for attempt in ... の break)。設計書 §3 の既存ロジックどおり |

### Sonnet が踏みやすい罠 (Stop condition §5 該当)

1. 「WinML が複雑なので stub に戻す」 → ❌ 設計書 §9 違反、Opus 委譲
2. 「windows-app がなくてもとりあえず動けばいい」 → ❌ §4.4 Path C で Opus 委譲
3. 「`register_execution_provider_library` が見つからないから panic」 → ❌ Result::Err で返す (L5)
4. 「`EnsureAndRegisterCertifiedAsync` の方がシンプルだから切り替える」 → ❌ registered EP 名取得ができない、個別 `FindAllProviders` 維持
5. 「thread_local が気持ち悪いから SelectedEp を `Arc<Mutex<...>>` で共有」 → ❌ 設計書 §3.3-3.4 のシンプルな thread_local 維持
